**Date:** 2026-06-09
**Scope:** Full-stack review of the application (Next.js 16.2 / React 19.2 / Supabase multi-tenant PWA) across features, security, data model, operations, testing, and launch readiness, followed by a day-by-day plan to reach production.

## Part 1 — 360 Review

### Executive summary

**Verdict: GO, with one operator prerequisite and a focused two-week hardening pass.**

The application is structurally production-ready. All 11 staff report flows and all 14 admin modules are implemented end-to-end (form → submit → DB → admin view → export/notification), with zero stub pages and zero TODO/FIXME markers in the codebase. CI gates lint, type-check, build, and run the cross-tenant RLS regression suite on every relevant PR. Auth, RLS (~98% table coverage), and security headers are mature and consistently applied.

No high-severity security findings were identified. What remains is hardening, not construction: a handful of medium-severity edge cases in the offline-sync path, RLS regression-test gaps for the core report tables, missing server-side observability (cron jobs can fail silently), offline-queue resilience, and a one-time Supabase migration-history reconciliation that must happen before automated migration delivery can run.

### What is already good (keep doing this)

- **Permission model**: flat `user_permissions` (view/submit/edit/admin per module), per-area permissions, role defaults, and SQL helper functions (`has_module_access` etc.) enforced at the DB level — not just in app code.
- **Defense in depth on dispatch**: `dispatch_rules_for_submission()` is SECURITY DEFINER and gated by facility + module-submit permission; the notification outbox is insert-locked to definer functions and service role.
- **Idempotent offline replay**: `onConflict: "local_id", ignoreDuplicates: true` upsert, employee resolved by `user_id` + `facility_id`, per-module `currentUserCan(…, "submit")` checks before persisting.
- **Typed failure contracts**: server actions return `{ ok: true } | { ok: false; error }` unions, so callers must handle errors.
- **CI breadth**: four workflows gate type errors, broken builds, RLS regressions, migration-prefix collisions, and cron-schedule drift before merge.
- **Build hygiene**: strict TypeScript, no eslint/ts suppression during `pnpm build`, CSP enabled in production.

## Part 2 — The 14-Day Plan

Two workstreams run in parallel: **code** (PRs through the existing CI gates) and **operator** (Supabase/Vercel/DNS tasks no PR can do). Days assume one primary engineer plus an operator with Supabase/Vercel access; everything is sized to fit comfortably with review time.

### Days 1–2 — Security hardening (code)

- **Fix F2**: in the offline-sync availability update branch, pre-fetch the targeted row and verify `employee_id` matches the authenticated employee before update; return 404 otherwise. (~10 lines.)
- **Fix F6**: add a `MAX_PAYLOAD_DEPTH` (e.g. 10) recursive check before handing the payload to module builders; return 400 on violation.
- **F9 audit**: run the RLS-enablement audit query against all public tables; if any business table lacks RLS, add a migration enabling it (next free `000000000001NN` prefix).

**Done when:** new vitest cases cover the ownership check and depth guard; RLS audit query returns zero unprotected business tables; CI green.

### Days 3–4 — RLS regression coverage (code)

- **Fix F3**: extend `supabase/tests/rls_isolation.sql` with, per core module (incidents, accidents, communications, refrigeration, air-quality, scheduling): Alice-reads-Facility-B → 0 rows; Alice-inserts-into-Facility-B → error; Alice's own-facility read/write succeeds.
- Add a negative dispatch-gate assertion: a user without `submit` on a module is denied by `dispatch_rules_for_submission()`.

**Done when:** `psql -f supabase/tests/rls_isolation.sql` passes locally against `supabase start`, and `rls-isolation.yml` passes in CI.

### Days 5–6 — Observability (code)

- **Fix F-OBS**: introduce a small structured-logging helper (JSON to stdout — Vercel's log drain picks it up) and use it in the three cron routes and the offline-sync route: start/end events with counts, error events with context (no secrets).
- Add error capture: either Sentry (server + client) or PostHog server-side events on cron completion/failure and server-action exceptions — pick one, not both.
- Emit a per-cron heartbeat event so a missed run is detectable (alert on absence, not just on error).

**Done when:** a forced failure in each cron route (e.g. bad service-role key locally) produces a structured error event visible in the chosen sink; happy-path heartbeats visible.

### Day 7 — Test-coverage parity (code)

- **Fix F4**: extract pure logic from `air-quality/_lib/submit.ts` and `incidents/_lib/submit.ts` into `compute.ts` modules (mirroring `reports/refrigeration/_lib/compute.ts`), keeping `server-only` imports out of the pure modules per `CLAUDE.md`.
- Add vitest suites (~30–50 cases each: parsing, validation, threshold/severity resolution, payload building).

**Done when:** `pnpm test` runs 8 module suites; no behavior change in the server actions (pure refactor).

### Days 8–9 — Offline-queue resilience (code)

- **Fix F5** in `public/sw.js`: exponential backoff between retries (`min(300, 2^retryCount) seconds`); record last error message per item; add a 30-day TTL purge for terminally failed items.
- Surface deadletter detail in `/reports/offline-queue`: show failed items with their last error and a force-retry action (via the existing `postMessage` protocol in `src/lib/offline/use-sync-queue.ts`).
- Bump the SW version/cache key so clients pick up the new worker.

**Done when:** manual test — submit offline, watch replay with backoff against a 500-returning endpoint, see the failure + error detail in the queue UI, force-retry succeeds once the endpoint recovers.

### Day 10 — Scale & product polish (code)

- **Fix F7**: add prev/next pagination (offset-based is fine at this scale) to the permissions, audit-log, and communications admin pages.
- **Decide F8**: remove the SMS toggle from `/account` (recommended for launch — reintroduce with a Twilio transport later) or wire a transport now if SMS is launch-critical.
- Quick pass on schedule-swap admin approval UI (F9) — verify approve/reject works end-to-end and is discoverable.

**Done when:** a facility with >500 users can page through permissions; no UI promises a channel that doesn't exist.

### Day 11 — Operator day (no code)

- **Fix F1** per `DEPLOY.md` §8: `supabase link` → `supabase migration list --linked` → `supabase migration repair --status applied <versions…>` for already-applied migrations (treat the grandfathered duplicate prefix `00000000000088` as covering both files; do not rename) → confirm alignment.
- Provision `SUPABASE_ACCESS_TOKEN`, `SUPABASE_DB_PASSWORD` (secrets) and `SUPABASE_PROJECT_REF` (variable) in GitHub; verify `deploy-migrations.yml` via `workflow_dispatch` does an **incremental, no-op** push.
- Provision Vercel env vars (all of `.env.example`); set `CRON_SECRET`; confirm Vercel cron fires all three routes with the bearer header (check the new heartbeat events from Day 5–6).
- Resend domain: SPF, DKIM, DMARC records for the `RESEND_FROM` domain; send a test email.
- Confirm Supabase backups/PITR are enabled on the production project.

**Done when:** `supabase migration list --linked` shows local/remote aligned; a dispatched migration workflow run succeeds as a no-op; all three crons produce heartbeats in production logs; test email lands in an inbox (not spam).

### Day 12 — Staging dress rehearsal

- Deploy the week's merged work to a staging/preview environment pointed at a staging Supabase project.
- Run the launch checklist (Part 3) end-to-end: login/logout/password flows, one submission per report module, an offline→online round-trip, an export (CSV + PDF), a routed notification (in-app + email), a retention purge dry-run, both themes, mobile/PWA install.
- Capture any defects; only fixes for checklist failures may merge after today.

### Day 13 — Docs, rollback plan, freeze

- Refresh `READINESS.md` (vitest now exists; light/dark both supported; fold in this review's resolved findings) and `DEPLOY.md` if Day 11 surfaced corrections.
- Write the rollback section: how to revert a Vercel deploy, how to roll back a migration (forward-fix policy recommended), who has access.
- Code freeze at end of day; tag the release candidate.

### Day 14 — Launch & monitor

- Production deploy (Vercel) + verify the migration workflow ran (or no-op'd) cleanly.
- Smoke test in production: login, one daily-report submission, one notification delivery, offline queue round-trip.
- Watch the observability sink for the first cron cycles; confirm heartbeats on all three jobs.
- Post-launch watch rotation for 48h; file post-launch follow-ups (Logbook rollout, screen-reader audit, cron rate limiting).
