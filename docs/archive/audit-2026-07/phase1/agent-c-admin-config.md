# Phase 1 Agent C — Admin Control Center Deep Audit (admin config → DB → staff module propagation)

Generated 2026-07-01. Read-only audit of all 10 assigned traces. Method: admin write action → Supabase table → staff-module read, flagging any hardcoded fallback that ignores admin config.

## Summary

The CLAUDE.md claim that forms are **metadata-driven holds up almost everywhere**. All eight report modules load their field/option config from per-facility Supabase tables, validate selections server-side against the same tables, and render explicit empty states (not hardcoded defaults) when config is missing. Permission and role changes take effect **without re-login** — every check is a per-request DB read (React `cache()` is render-tree-scoped only; nothing is stored in the JWT/session).

The gaps are at the edges, not the core:

1. **Module toggles don't reach the dashboard.** `facility_modules` hides sidebar/tab-bar entries but the dashboard tile grid ignores it, so a "disabled" module remains one tap away (C-01). Route access is intentionally not gated by the toggle (documented as visibility-only).
2. **Two soft-enforcement seams**: ice-operations submit doesn't re-check `enabled_operation_types` server-side (C-02), and staff my-schedule hardcodes Sunday weeks while the rest of scheduling honors `schedule_settings.week_start_day` (C-05).
3. **One orphaned config surface**: admin-managed `incident_types` are never written to any report — the staff form has no type field and no admin action assigns one, so the admin "Types" tab and its history filter are dead config (C-04).
4. **A three-way fallback disagreement** in ice-depth thresholds (DB default 0.99/1.75 vs staff fallback 1/1.5 vs admin display fallback 1/2) that only bites when the settings row is missing (C-03).
5. A handful of Low-severity hardcoded lists that are either deliberate (DB-check-constrained enums, the SVG body diagram) or documented sync-required constants (daily-areas cap of 30).

Counts: **0 High, 5 Medium, 13 Low.** No case was found where a staff form silently substitutes hardcoded options for existing admin config.

## Verified OK

- **Daily reports**: areas/templates/checklist items 100% DB-driven (`reports/daily/page.tsx:61-120`); no hardcoded fallback areas/items; area access enforced at **three layers** — render filter (`actions.ts:39-72`), submit action (`_lib/submit.ts:106-116` rejects without `module_area_permissions.can_submit`), and RLS (`has_area_submit_access`, migration 90) — a staff user cannot submit to an unassigned area; 30-area cap enforced by DB trigger (client constant is UI-only); business date derived from facility timezone.
- **Refrigeration**: all CRUD → `refrigeration_sections/_equipment/_fields/_thresholds` (`admin/refrigeration/actions.ts:106-832`); staff `formSections` fully DB-assembled, zero hardcoded fields; `NormalRangeHint` resolves thresholds from DB with per-equipment override precedence and correct °F↔°C conversion; empty config renders an explicit error, not defaults.
- **Air quality**: reading types, tiers, and per-facility overrides fully DB-driven; range badges/tier hints computed from DB-effective tiers; **override loosening rejected server-side** — `validateOverrides` (`reports/air-quality/_lib/compliance.ts:243`) enforced at `admin/air-quality/actions.ts:825-828` and unit-tested (`compliance.test.ts:165-180`). (An earlier sub-trace claimed this validation was missing; code inspection disproves it.)
- **Ice operations**: staff tabs derive from `enabled_operation_types` with documented fail-open (empty = all; staff `types.ts:99-106`); disabled type's tab disappears and direct URLs redirect (`[operationType]/page.tsx:147-157`); `DEFAULT_OPERATION_TYPE` redirect cannot dead-end; rinks/equipment/fuel types/circle-check items+templates all per-facility, active-only, with empty states.
- **Ice depth**: layouts/points/rinks/settings fully DB-driven with `notFound()`/empty-state handling; severity computed server-side by pure, unit-tested `severityFor()`; thresholds+unit **snapshotted per submission** (immune to later admin edits); severity colors admin-configurable; DB caps (8 layouts / 60 points) trigger-enforced.
- **Incidents**: severities/activities/spaces DB-driven with "Not configured yet" hard-block when no severities; submit re-validates every selected id against facility + `is_active` (`_lib/submit.ts:63-109`) — arbitrary ids rejected.
- **Accidents**: all five dropdown categories from `accident_dropdowns` via per-facility cached read with tag invalidation (`_lib/dropdowns.ts`); workers-comp instructions from `accident_workers_comp_settings` (page.tsx:194); severity→alert mapping unit-tested.
- **Lists module**: single governed domain (`facility_timezone` → `facility_dropdown_options`), consumed by the facility settings form; `TIMEZONE_OPTIONS` fallback is a documented bootstrap for pre-seed facilities; no orphaned list domains and no staff dropdowns bypassing it.
- **Module toggles**: server action admin-guarded with session-resolved facility + RLS; fail-open on missing config is deliberate and documented ("a missing config can never hide the entire app"); `revalidatePath` on the staff layout roots makes toggles effective on next navigation without re-login.
- **Roles & permissions**: no permission data in JWT/session (`grep` for `app_metadata`/`jwt` claims: zero hits); `getCurrentUser`/`requireAdmin` are per-request DB reads wrapped in render-tree-scoped React `cache()`; `currentUserCan` calls the `current_user_has_permission` RPC per call, fail-closed; matrix writes upsert `manual_override` rows and `revalidatePath`; role assignment guarded by `canAssignRoleLevel` (non-supers cannot mint admins) with role defaults re-seeded via `apply_role_permission_defaults` preserving manual overrides. **All changes effective on the next request — no re-login required.**
- **Scheduling**: `updateSchedulingSettings` validates ranges (week_start_day 0..6 etc.) and upserts `schedule_settings`; admin grid, month grid, warnings engine, and the DB RPC violation engine all read settings server-side; per-employee `max_weekly_hours` cap and cert hard-blocks enforced in server actions (`grid-actions.ts` + `scheduling_assignment_violations` RPC), with `block_on_violations` opt-in read from settings; staff availability honors `week_start_day`; job areas DB-driven in both admin grid and staff views.
